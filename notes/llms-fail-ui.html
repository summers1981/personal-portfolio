Why LLMs fail at UI — and what that reveals about how we document components

UX October 2025 · 4 min read

Ask an LLM to build a button component and it'll produce something reasonable. Ask it to build a button that follows your design system's exact conventions — using your tokens, respecting your usage rules, fitting correctly into your component hierarchy — and it will almost certainly get something wrong.

The usual response to this is to blame the model. It's not smart enough, not trained on the right data, not capable of understanding design nuance. That explanation is both partially true and almost entirely unhelpful — because the thing that would actually fix the problem isn't a smarter model. It's better documentation.

"LLM failure at UI is usually a documentation failure. The model can't follow rules it was never given — and most component docs were never written with a machine reader in mind."

What models actually see

When a model generates UI, it's working from whatever context is available: the prompt, any documentation passed in, the structure of the codebase it can see, and its training data. For most design systems, that context is sparse and poorly structured.

Component documentation is typically written as prose — helpful for a human who can read between the lines, fill in obvious gaps, and apply contextual judgment. A model has none of that. It takes the documentation literally, and when the documentation is incomplete or ambiguous, it fills the gaps with its best guess from training data — which is to say, from the averaged conventions of every UI library it's ever seen, not yours specifically.

The result is UI that's plausible but wrong. Components used in contexts the system doesn't intend. Token names referenced that don't exist in your system. Composition patterns that are locally coherent but inconsistent with the product's actual design language.

The documentation gap

Most component documentation contains some combination of the following:

  • A visual example (useful for humans, invisible to text-based models)
  • A prose description of what the component does
  • A list of props with types and defaults
  • Sometimes: usage guidelines written for designers
  • Rarely: explicit statements of what this component should not be used for
  • Almost never: structured metadata about relationships to other components, semantic purpose, or usage constraints in machine-parseable form

This is documentation for humans. It works because humans bring context. They know what a "primary button" means because they've seen hundreds of products. They know the difference between a "destructive action" and a "primary action" because they understand product intent.

Models don't have that context unless you give it to them. Which means the documentation gap is actually a context gap — and it's one that design system teams can close.

What machine-legible documentation looks like

The components that LLMs handle well share a few properties. Their purpose is explicit, not inferred. Their constraints are stated, not assumed. Their relationships to other components are structured, not described in prose. And their usage rules are specific enough that there's only one reasonable interpretation.

In practice, this means adding structured metadata alongside human-readable docs. A short, unambiguous purpose statement: "Use for the single primary action in a view. One per screen." A list of forbidden combinations: "Do not use alongside Button.secondary as a co-equal action — use with a text link instead." A semantic role declaration: "Interactive, confirmatory, high emphasis."

None of this replaces human documentation. It supplements it with a layer that's optimised for a different reader.

The mirror this holds up

The interesting thing about auditing documentation for AI legibility is what it reveals about the documentation itself. Components with ambiguous purpose are ambiguous to models because they're ambiguous, full stop — the AI is just less willing to paper over the ambiguity. Usage rules that feel obvious to the team are only obvious because of shared context that new team members also lack.

Making component documentation machine-legible tends to make it human-legible too — clearer, more explicit, more trustworthy. The LLM failure is a useful diagnostic. It shows you where your system's intent isn't written down.
